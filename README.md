# Soundgram — Frontend тестовое задание

Веб-приложение в формате Telegram Mini App для обмена музыкальными предпочтениями.

## Запуск

```bash
# Требуется Node.js 20+
npm install
npm run dev
```

Откроется на `http://localhost:5173`

## Технологии

- **Vite + React 19 + TypeScript** — сборщик и стек
- **React Router v6** — навигация между экранами
- **Plain CSS с CSS-переменными** — стили без UI-библиотек

## Структура

```
src/
  components/
    TabBar         — нижняя панель навигации
    Post           — карточка поста с треком в ленте
    PlayerSheet    — bottom-sheet плеер с Яндекс.Музыки iframe
    FollowersModal — bottom-sheet список подписок
  pages/
    Feed           — лента постов (основной экран)
    Search         — поиск
    Profile        — профиль пользователя
  data/
    mockData       — захардкоженные данные (15 постов, 7 пользователей)
```

## Ключевые решения и допущения

### Плеер как bottom-sheet
При нажатии «Слушать» снизу экрана выезжает панель с iframe-плеером Яндекс.Музыки.
Это позволяет не терять контекст ленты во время прослушивания.
Одновременно активен только один плеер — повторный тап на тот же трек закрывает панель.

### Стейт плеера локален для Feed
`activeTrack` живёт внутри компонента Feed, а не в глобальном стейте.
При переходе на другую вкладку плеер закрывается сам — пользователь не продолжает
слушать трек незаметно для себя.

### Mobile-first без overflow на body
Контейнер ограничен `max-width: 480px`, `overscroll-behavior: none`.
Скролл реализован только внутри `.feed-list`, а не на странице — приложение
ощущается как нативное, без лишних bounce-эффектов.

### Допущения
- Backend не реализован, все данные захардкожены в `mockData.ts`
- Экран «Поиск» имеет вёрстку без поискового функционала (по условию задания)
- Медиатека в профиле — заглушка
- Список подписок и подписчиков использует один и тот же mock-список
